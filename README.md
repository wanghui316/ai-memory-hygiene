## 🔥 为什么你需要它（应用价值）

如果说这套方法论只解决「整齐」问题，那就是个普通文件管理技巧。
实际上它解决的是 **AI 应用的算力焦虑**。

### 📌 韬定律的启发

2026 年 5 月，华为何庭波提出韬（τ）定律，本质是 **STCO（System Technology Co-Optimization）** 方法论的具体演进——当晶体管几何缩微逼近极限，不再追求尺寸缩小，而是系统性压缩信号传播的时间常数（τ）。

本方法论与韬定律同构：面对 LLM 上下文堆叠的极限，我们不再追加信息容量，而是**优化信息被检索的时间**。
从 “信息-系统协同优化” （Information-System Co-Optimization）这一一般范式看，两个领域共享同一思想内核——**放弃单一维度的资源堆叠，转向多层级协同优化**。

### 问题现状

LLM 推理本质是「条件概率计算」—— 每次都要把上下文重新跑一遍 Attention。
后果：
- 📈 **Token 成本指数增长**：上下文越长，每次调用越贵
- 🐢 **延迟劣化**：长 prompt 下首 token 延迟显著增加
- 📵 **端侧部署几乎不可能**：手机、IoT 跑不动大模型
- 💸 **服务器账单爆表**：Agent 长任务成本失控

### 本方案的解法

**「把耗算力的部分，提前到索引构建阶段做一次；运行时只做廉价指针查找。」**

用一句话重新表述核心思想：

> **MEMORY.md = 目录 + 铁律；不是「整齐」，是「把运行时计算从 O(N) 降到 O(1)」**

### 量化收益（实测估算）

| 场景 | 传统方案 token 消耗 | 本方案 token 消耗 | **节省** |
|---|---|---|---|
| 查「当前持仓」× 100 次（数据 10K token） | 1,000,000 | 20,000 | **98%** |
| 查「当前持仓」× 100 次（数据 100K token） | 10,000,000 | 20,000 | **99.8%** |
| 查「历史规则」× 100 次（数据 1M token） | 100,000,000 | 20,000 | **99.98%** |

**核心结论：数据量越大，节省越多。**

### 与同类方案的对比

| 方案 | 思路 | 算力节省 | 运维成本 | 时间戳感知 |
|---|---|---|---|---|
| **传统全量 Prompt** | 暴力塞上下文 | ❌ 0% | 中 | ❌ |
| **向量数据库 RAG** | 语义检索 | ⭐⭐ 70% | 高 | ❌ 易返回过期 |
| **MemGPT** | 虚拟上下文管理 | ⭐⭐⭐ 80% | 高 | ❌ |

### 🎓 面向大模型开发者的应用场景

对于**自己微调 / 训练 / 蒸馏模型**的开发者，这套方法论提供了新的思路：

**1. 训练数据治理**
- SFT/RLHF 数据集按业务主题分目录（如「代码生成」「问答」「总结」）
- 每个数据集配 `_index.md`，记录样本数、版本、最后更新
- 训练时按索引取数，避免重复加载整个数据集

**2. 评测集管理**
- 不同模型版本的评测结果按日期归档
- `_index.md` 标出「最佳模型」「历史对比」
- 防止评测「张手李手」拿错数据

**3. 模型 checkpoint 治理**
- 大模型 checkpoint 占空间（几个 GB 到几百 GB）
- 只保留「最佳 + 最新 + 关键节点」三套，其余打 tag 归档
- 推理时按需加载，避免同时塞满显存

**4. 提示词工程（Prompt Engineering）**
- 提示词版本管理（v1 / v2 / v3），附带效果指标
- `_index.md` 快速定位「当前生产环境在用哪版」
- A/B 测试时按版本切换，不混乱

**5. 多模型路由**
- 同一业务有多个模型选择（GPT-4o / Claude / 本地 Llama）
- 按「成本/效果」路由，简单 query 走小模型、复杂 query 走大模型
- 决策逻辑写在 `_index.md` 里，一目了然

**6. 微调数据采样**
- 从原始数据池中按主题、版本、覆盖率采样
- 采样规则写成 `_index.md`，复现性 100%
- 避免「上次怎么调的来着？」的困境

**理论路径**：

```
原始数据池（按主题分目录）
   ↓
采样索引（_index.md 记录规则）
   ↓
训练数据（按版本归档）
   ↓
Checkpoint（只保留关键节点）
   ↓
评测（按日期归档结果）
   ↓
部署路由（索引驱动）
```

**示例场景**：
- 训练 LoRA 时：以前要加载全部数据 → 现在按主题分批
- 评测新模型时：以前翻历史文件 → 现在看 `_index.md` 找「最佳基线」
- 上线生产时：以前「谁改了 prompt？」→ 现在看 `_index.md` 找到当前版本

```python
# 伪代码：用 HygieneMem 方式管理训练数据
def load_training_data(task):
    # 1. 加载 MEMORY.md（恒定 token）
    rules = load("MEMORY.md")
    
    # 2. 查 _index.md 找当前推荐的数据版本
    index = load(f"datasets/{task}/_index.md")
    recommended_version = index["current"]
    
    # 3. 按需加载（而非全量加载）
    data = load(f"datasets/{task}/{recommended_version}.jsonl")

def load_all_data_legacy(task):
    return load(f"datasets/{task}/all_versions.jsonl")  # 几 GB
```

## 🎯 这是什么

用 AI 助手久了，你一定遇到过这些问题：

- **记忆越来越长**：MEMORY.md 从 1KB 涨到 50KB，AI 启动变慢、回答越来越慢
- **找不到「哪个是最新的」**：旧版本和新版本堆在一起，AI 经常拿过时数据回答
- **不同任务的数据混在一起**：股票分析、论文写作、工作交接全塞一个文件
- **不敢清理**：清掉又怕丢关键规则，AI 失去上下文后行为失常
- **新会话失忆**：昨天聊的项目，今天 AI 完全不记得

**这套方法解决的就是这些问题。**

核心思想一句话：**MEMORY.md = 目录 + 铁律，详细内容各回各家**。

## 🧠 核心理念

```
MEMORY.md     → 身份 + 长期规则 + 目录索引（≤100 行，常驻上下文）
业务目录/     → 详细内容（每个目录配 _index.md 做导航）
memory/       → 当日要点 + 决策指针（≤30 行/文件）
tmp/          → 一次性产物（可随时清）
```

**关键原则：**

| 原则 | 解释 |
|---|---|
| **分层存放** | 规则、详细内容、当日指针、物理临时文件各归各位 |
| **索引而非堆叠** | 每个目录配 `_index.md`，标日期 + 指针，AI 自己去找最新文件 |
| **指针而非副本** | memory/ 只写"做了什么 + 指向哪"，不复制业务目录的详细内容 |
| **定期清理** | tmp/ 随时清、work_records/ 每月清、MEMORY.md 季度体检 |
| **自定义指标必须说明** | 报告中自己造的评级/打分要附定义，避免被读不懂 |

## 📜 三大铁律

### 铁律 1：身份与规则放 MEMORY.md

只放三类内容：

```
## 0. 身份（机器人 + 服务对象 + 时区）
## 1. 交流铁律（回复风格 + 做事顺序 + 执行原则）
## 2. 业务目录索引（每个业务目录 + 一句话说明）
## 3. 写作风格（如果长期不变）
## 4. 已安装技能列表（如需要）
## 5. 清理节奏（提醒定期清理）
```

**绝对不放：**

- ❌ 聊天记录、对话摘要
- ❌ 当前持仓、最新分析（变动数据）
- ❌ 详细的 API 密钥（放 TOOLS.md，需要时才读）
- ❌ 超过 100 行的内容

### 铁律 2：详细内容按业务分类

按你**实际的工作主题**分目录，而不是按文件类型。

**错误示范**：

```
❌ /md_files/
❌ /reports/
❌ /data/
```

**正确示范**：

```
✅ /股票/        # 所有跟股票相关的都在这里
✅ /research/    # 所有跟论文相关的都在这里
✅ /work_records/  # 所有跨会话任务交接都在这里
```

每个业务目录配 `_index.md`，AI 找数据时先扫索引：

```markdown
# 股票目录索引

## 当前状态指针
- 最新持仓：`持仓跟踪/今日状态（2026-07-02）_v1.md`
- 最新复盘：`复盘/复盘报告_2026-07-02.md`
- 当前规则：`规则档/V4.4.5_交易规则_20260702.md`

## 历史
- V4.4.4 (2026-07-01)
- V4.4.3 (2026-07-01)
```

### 铁律 3：memory/ 只放指针

memory/ 是**当天发生了什么**的极简记录，不写详细内容。

**示例 `memory/2026-07-05.md`：**

```markdown
# 2026-07-05

## 今天做了什么
- 整理工作区目录（详见 work_records/2026-07-05_目录重构.md）
- 股票分析三主题（详见 work_records/2026-07-05_股票分析专题.md）

## 关键决策
- 启用新目录结构
- 删 MEMORY_旧版备份.md

## 当前生效
- 规则：V4.4.5
- 持仓快照：7-02
```

**不写：**

- ❌ 详细数据表格
- ❌ 完整分析内容
- ❌ 超过 30 行

## 📊 自定义指标的说明义务

**这是新手最容易踩的坑。**

你在报告中用了自己造的评级，比如：

> 「海光信息：弹性=极高」

读者（包括 AI 助手自己过几天回看）会问：
- 「弹性」是什么？
- 怎么算出来的？
- 极高是多高？

**正确做法：附简要定义（三要素）**

```markdown
### 弹性指标说明
- **含义**：股价潜在涨幅空间，由业务弹性、估值弹性、题材弹性综合评估
- **计算依据**：
  - 业务弹性：主业增速、订单/产能扩张速度
  - 估值弹性：当前估值 vs 业绩预期
  - 题材弹性：政策/技术/事件催化想象空间
- **取值范围**：极高 / 高 / 中等 / 偏低 / 低（五档）
```

**其它常见自定义指标：**

- 风险等级
- 估值水位
- 情绪热度
- 信号强度
- 业绩确定性

每个第一次出现时都加一行说明。

## 🧹 清理节奏

| 目录 | 清理频率 | 触发条件 |
|---|---|---|
| `tmp/` | 随时 | 用户说「清 tmp」 |
| `work_records/` | 每月 | 30 天前的过期记录 |
| `memory/` | 每周 | 合并/清理过期条目 |
| `股票/复盘/` | 月底 | 归档到 `复盘归档/YYYY-MM/` |
| `MEMORY.md` | 季度 | 体检：删除已失效规则 |
| 业务目录内 | 按需 | 新版本出来后旧版可归档 |

**清理原则：**

- 删除前确认「业务目录里有完整版」
- 保留至少一份历史快照（如月度归档）
- MEMORY.md 体检时删「已不再适用」的规则

## 🚀 迁移指南（从「一团乱」到「清爽」）

如果你已经在用 AI 但记忆管理混乱，按这四步走：

### Step 1：备份旧的 MEMORY.md

```bash
cp MEMORY.md MEMORY_旧版备份.md
```

不要急着删，先备份到 `backups/` 目录。

### Step 2：提取长期规则

问自己这些问题：

- 「这个规则一周后还会生效吗？」
  - 是 → 留 MEMORY.md
  - 否 → 搬到业务目录
- 「这是身份信息还是变动数据？」
  - 身份（我叫什么、用户是谁、时区）→ MEMORY.md
  - 变动（当前持仓、最新分析）→ 业务目录

**重构后的 MEMORY.md 模板：**

```markdown
# MEMORY.md

## 0. 身份
- 机器人：XXX
- 用户：YYY
- 时区：Asia/Shanghai

## 1. 交流铁律
- 精简直接
- 任务完成一次性汇报
- 多步任务只给结果

## 2. 业务目录索引
- 股票 → `股票/`
- 论文 → `research/`
- 工作交接 → `work_records/`

## 3. 写作风格
- 排比递进、四字结构

## 4. 已安装技能
- `skills/` 目录

## 5. 清理节奏
- tmp/ 随时清
- work_records/ 每月清
```

### Step 3：按主题建业务目录

```
mkdir -p 股票 research work_records memory tmp
```

把原来的散落文件按主题分类移动到对应目录。

### Step 4：建 _index.md

每个业务目录配一个：

```markdown
# 业务目录索引

- 最新 XXX：`路径/文件名.md`
- 当前规则：`路径/文件名.md`

## 历史
- v1 (YYYY-MM-DD)
- v2 (YYYY-MM-DD)

## 操作流程
1. 用户问 X → 打开哪个文件
2. 用户问 Y → 打开哪个文件
```

## 🔧 工具适配

这套方法适用于任何支持文件系统的 AI 助手：

### OpenClaw（最契合）

- 内置 `MEMORY.md` + `memory/` 加载机制
- `skills/` 目录自然契合业务目录
- `cron` + `heartbeat` 提醒清理

### Claude Code / Cursor / Windsurf

- 项目根目录的 `CLAUDE.md` / `.cursorrules` 对应 MEMORY.md
- 子目录结构完全适用
- 用 `@filename` 引用文件时 AI 会自动读取

### Cherry Studio / ChatGPT

- 每次对话粘贴 MEMORY.md 内容
- 用「对话模板」功能预设索引结构
- 长任务拆成多段，每段开头粘索引

### 其他 AI（通用原则）

- 把 MEMORY.md 内容放在系统提示或开头
- 业务目录文件用工具调用读取
- 不要把全部内容塞进一个 prompt

## 🎓 设计哲学

### 为什么不把全部内容都放 MEMORY.md？

1. **Token 成本**：MEMORY.md 每次启动都加载，50KB 就是每次都付钱
2. **注意力分散**：AI 会在海量内容里"迷失"，反而抓不住重点
3. **更新困难**：改一处牵动全身，不如分层修改局部

### 为什么需要 _index.md？

AI 不是神，没法扫描整个目录判断哪个最新。

给它**显式的指针**（"最新在 XXX"），它就能：
- 1 秒找到
- 不会读错版本
- 不会瞎猜

### 为什么 memory/ 不直接写详细内容？

- **流动性**：今天的"重要"，明天就过时
- **检索性**：指针比副本好找
- **清理性**：删起来不心疼

## 🛠 CLI 工具：mh (memory-hygiene)

将方法论固化为可执行的代码工具，核心算法为**上下文预算分析**。

### 安装

```bash
# 直接运行
python3 bin/mh.py status

# 或安装到系统
pip install -e .
mh status
```

### 命令

| 命令 | 功能 | 示例 |
|------|------|------|
| `init` | 初始化工作区目录结构 | `mh init --dir ~/workspace` |
| `status` | 分析工作区健康状况 | `mh status` |
| `check` | 检查约束违例 | `mh check --json` |
| `index` | 更新 _index.md 导航 | `mh index --dir ./trading` |
| `prune` | 列出可清理的文件 | `mh prune` |

### 核心算法

**Context Budget Analyzer（上下文预算分析算法）**：

1. 扫描工作区所有 `.md` 文件
2. 按内容类型（中文、英文、代码、表格）估算 token 消耗
3. 对照方法论铁律（MEMORY.md ≤100行、日记 ≤30行等）逐条检查约束
4. 计算 0-100 健康评分
5. 生成优先级排序的优化建议

输出示例：

```
📊 健康评分: 71/100
    [█████████████████████░░░░░░░░░]

📝 Token 预算:
    MEMORY.md:          1,455 tokens
    memory/ 日记目录:  22,782 tokens
    _index.md 导航:       420 tokens

⚠️ 发现 3 个问题:
    🟡 memory/2026-07-03.md 行数 56 超过 30 行上限
    🟡 目录 skills/ 缺少 _index.md
```
